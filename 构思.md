仿照 Bookstore-2025，对 Ticket-System-2026 进行构思。

考虑到这次不再出现 Invalid 情况，checker 不再被需要。

以下的 bpt 是基于文件读写的 B+ 树（模板类），维护一个 pair（按两个关键字先后排序），已经实现了插入，删除，查找所有 .first 和给定 first 相同的值三个功能。可以增加必要的简单功能。

### 职责划分

- interactor.h 负责和外界进行交互。由于不再需要 checker，interactor 要负责读取具体的指令并把数据传给对应系统。它不需要任何存储具体信息。

- account.h 负责帐户管理。它需要 bpt<username, pos>，其中 pos 指向（通过另一个 bpt:accountmemory<int,tuple>) tuple(password, name, mailAddr, privilege) (也即 tuple(char[30], personname, char[30], int)) 来存储**账户名、密码、姓名、邮箱、优先级**。由于 username 唯一所以不会有问题。一个 bpt:account 存储所有用户列表，一个 bpt:stack 存储当前用户列表。

- train.h 负责火车管理。它需要 bpt<trainID, pos>，其中 pos 指向（通过另一个 bpt:trainmemory<int,tuple>） tuple(stationNum, stations[], seatNum, prices[], startTime, travelTimes[], stopoverTimes[], saleDate_start, saleDate_end, type, remaining_tickets[max_date][max_num]) (也即 tuple(int, stationname[100], int, int[100], (daily time) int, int[100], int[100], (monthly time) int, (monthly time) int, char, int[100][100]))。由于 trainID 唯一所以不会有问题。一个 bpt:hidden 存储还未 release 的车，一个 bpt 存储已经 release:released 的车。

- order.h 负责订单管理。它需要 bpt<username / trainID, pos>，其中 pos 指向（通过另一个 bpt:ordermemory<int,tuple>） tuple(timestamp, trainID, username, date, start, end, number, state) (也即 tuple(int, char[20], char[20], (monthly time) int, stationname, stationname, int, 1(success)/2(pending)/3(refunded)) )。前者 (userorder) 用来回答 query_order，后者 (queue) 用来管理等待补票的队列。

实际上还需要一些别的东西来维护，一会写到了再说。

- 为了 query_ticket 和 query_transfer，需要在 train 中引入新的 bpt<stationname, trainID>。称为 stationer。


### 指令处理

- [N] add_user 

在 stack 中找到 -c（找不到就 fail），检查 -u 在 account 中找不到（否则 fail），检查 -c 权限大于 -g（否则 fail），在 accountmemory 中 insert，加入 account 中。特判第一次。

- [F] login

在 account 中找到 -u（找不到就 fail），检查 -p 等于密码（否则 fail），检查 stack 中找不到（否则 fail），加入 stack 中。

- [F] logout

在 stack 中找到 -u（找不到就 fail），把它删掉。

- [SF] query_profile

在 account 中找到 -c 和 -u（找不到就 fail），在 stack 中找到 -c（找不到就 fail），检查 -c = -u 或者 -c 权限大于 -u（否则 fail），输出 -u 的 username，name，mailAddr 和 privilege，一个空格隔开。

- [F] modify_profile

在 account 中找到 -c 和 -u（找不到就 fail），在 stack 中找到 -c（找不到就 fail），检查 -c = -u 或者 -c 权限大于 -u（否则 fail），检查 -g 小于 -c 权限（否则 fail），找到新的 profile，在 accountmemory 中更改 -u 的 profile。

- [N] add_train

在 hidden 和 released 中找 -i（找到则 fail），在 trainmemory 中加入这个 train，在 hidden 中加入。

- [N] delete_train

在 hidden 中找 -i，找到则删除，否则 fail。

- [N] release_train

在 hidden 中找 -i（找不到则 fail），转到 released 中，在 stationer, timer 和 coster 中加入。

- [N] query_train

在 hidden 或者 released 中找 -i（找不到则 fail），找到了直接依次输出。

- [SF] query_ticket

在 stationer 里把经过 -s 和 -t 的 trainID 都拿出来，过滤掉不开的车，取交，要求先过 -s 再过 -t，把合法的拿出来排序，查余票。

- [N] query_transfer

在 stationer 里把经过 -s 和 -t 的 trainID 都拿出来，维护两个 map<stationname, vector< tuple< arrival_time, time, cost, trainID> >> / vector< tuple< arrival_time, cost, time, trainID> >> 分别为 tmap, cmap。先扫 -s，检查日期是否合法，如果合法，扫之后经过的点更新 tmap 和 cmap。再扫 -t，检查日期是否合法，如果合法，扫之前经过的点，利用 tmap 和 cmap 更新答案，具体地说，把对应的 vector 拿出来检查。输出即可。

- [SF] buy_ticket

在 stack 里找到 -u（找不到就 fail），在 released 里找到 -i（找不到就 fail），用指针拿到 ordermemory 中对应的返回值，找到 -d 对应的 int[100]，直接暴力 check 够不够。不够的话就根据 -q 判断要不要扔进 queue 里面。然后把这个订单扔进 userorder 里面，修改 ordermemory 中的余量。

- [N] query_order

在 stack 里找到 -u（找不到就 fail）。直接从 userorder 里面找到所有订单倒着输出。

- [N] refund_ticket

跑一遍 query_order 做的事情，找到要退回的订单的信息。在 released 里找到 -i（找不到就 fail），用指针拿到 ordermemory 中对应的返回值，找到 -d 对应的 int[100]，退票但不急着扔回去。然后根据 trainID 拿出来 queue 中所有可能的合法订单，依次检查，合法的就从 queue 里删掉并改 userorder。最后扔回 ordermemory 里面。

- [R] clean

清空所有 bpt 对应的文件。

- [R] exit

清空 stack 对应的文件。（后续还可以接着操作）
